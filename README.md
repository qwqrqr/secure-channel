# Secure Channel — стенд защищённого канала связи HQ ↔ Remote Office

Дипломный проект: «Проектирование защищённых каналов связи с удалённым офисом».

Стенд демонстрирует работу VPN-подобного канала между двумя офисами с использованием:
- **ECDH P-256** для обмена ключами
- **HKDF-SHA256** для деривации сеансовых ключей
- **AES-GCM-256** для шифрования полезной нагрузки
- **HMAC-SHA256** для целостности и аутентичности (Encrypt-then-MAC)
- **Counter-based anti-replay** для защиты от повторов
- **DLP-фильтр** до шифрования — защита периметра

## Запуск стенда

### Шаг 1. Установить зависимости

```bash
cd gateway
npm install
```

### Шаг 2. Запустить шлюз и статический сервер (два терминала)

Терминал 1:
```bash
npm start
```
Запустит:
- WebSocket шлюз: `ws://localhost:8080`
- HTTP аудит: `http://localhost:8081`

Терминал 2:
```bash
npm run static
```
Запустит статический сервер для клиента: `http://localhost:3000`

### Шаг 3. Открыть стенд

В браузере:
1. Главная: `http://localhost:3000` — описание стенда и быстрые ссылки.
2. **HQ**: `http://localhost:3000/office.html?peer=hq` — открыть в одной вкладке.
3. **Remote**: `http://localhost:3000/office.html?peer=remote` — в другой вкладке.

В обеих вкладках:
- Указать одинаковый ID туннеля (по умолчанию `office-2026`).
- Нажать «Подключиться к шлюзу».
- Через секунду стороны автоматически проведут ECDH-handshake.
- Должен появиться зелёный статус `ESTABLISHED` и одинаковый отпечаток сеансового ключа.

## Сценарии для защиты диплома

### 1. Демонстрация установки канала
Открыть HQ и Remote → нажать «Подключиться» → показать в журнале:
- `peer_joined` → `tunnel_both_present` → `handshake_relay` → `tunnel_established`
- Совпадение fingerprint у обеих сторон.

### 2. Передача защищённого сообщения
В HQ ввести «Привет, это секретное сообщение», нажать «Зашифровать и отправить».
В Remote сообщение появится в чат-истории расшифрованным.

### 3. Демонстрация шифрования (Eve)
Открыть «Перехватчик» — там виден сырой шифротекст:
```
{ iv: "...", ciphertext: "...", hmac: "...", counter: 1 }
```
Никакого открытого текста, даже шлюз не может расшифровать.

### 4. Атака подмены (MITM)
Нажать «Атака: подмена пакета» → пакет с искажённым ciphertext уходит на ту сторону →
HMAC-валидация отвергает пакет → в журнале появляется
`АТАКА ПЕРЕХВАЧЕНА: HMAC_INVALID: пакет был изменён или подделан`.

### 5. Защита от replay
Через журнал шлюза `/audit` показать поле `counter`. Объяснить, что при попытке
повторной отправки того же пакета шлюз отвергнет с `replay_detected`.
Подтверждение работы — `npm run test:replay`.

### 6. DLP-фильтр
В HQ ввести строку, содержащую ИИН (например, «Мой ИИН 901231400123») →
кнопка «Зашифровать и отправить» → DLP блокирует ещё до шифрования.
Показать, что сообщение **не уходит** в канал.

## Что игнорируется в учебном стенде (важно для защиты)

В реальной системе дополнительно нужно:
- **Аутентификация сторон при handshake** — иначе атакующий может подменить
  публичный ключ. Решается сертификатами PKI или pre-shared keys.
- **Forward Secrecy** — текущая схема даёт PFS только если генерировать
  ECDH-пару заново на каждый сеанс (что и сделано).
- **TLS поверх WebSocket** (wss://) — в продакшне обязательно.
- **Хранение DLP-правил и логов в защищённой БД** с RBAC.

Это можно вынести в раздел «Перспективы развития» дипломной работы.
